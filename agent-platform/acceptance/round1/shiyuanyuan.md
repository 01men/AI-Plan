- 日期：2026-07-19 ｜ 环境：http://127.0.0.1:8000（本地部署）｜ person_id=3，tier=coach
- 角色目标：验证「立项 → 派活 → 审核 → 激励」全链路是否顺，能否支撑我作为数字化项目负责人日常推进 65 个数字员工的扩围落地。
- 并发指标：全程 `database is locked` 发生 **0 次**。

## 一、旅程逐步结果

### ① 驾驶舱看覆盖率与动态流 — ✅
- 预期：一眼看到覆盖率等核心指标与最新动态。
- 实际：首屏 7 张 KPI 卡（场景覆盖率 6.2% 目标≥70%、验收通过率、活跃使用率 7.7%、累计节省工时 185.4h、交付准确率 96.2%、年化综合收益 ¥79万、Skill 复用数 13），每张都带目标值，红黄绿自明；投入 vs 收益条形图、近 14 天趋势、四波次推进、TOP8 榜单齐全。**动态流（"心跳动态流"）存在但位于页面最底部**，1440×900 首屏完全看不到，需滚动两屏以上。证据：`shiyuanyuan_dashboard.png`、`shiyuanyuan_dashboard_full.png`。
- 独立完成：能，但"动态流"是找了两屏才发现的。

### ② 场景库把"待立项"场景敏捷立项 — ⚠️（立项顺，数字员工断链）
- 预期：一键立项，自动建工作区**且数字员工已入区**。
- 实际：
  - 场景库列表每行右侧有绿色「敏捷立项」按钮，点一次即完成（1 步，无多余确认）。API `POST /api/scenarios/6/initiate` 返回 200，场景 6「经营数据日报自动生成」状态→已立项，自动创建工作区 #7「项目·经营数据日报自动生成」（created_by=3）。✅
  - **但 76 个待立项场景的"数字员工"列全部显示"未绑定"**（`GET /api/scenarios` 确认 agent_id 全为 null），立项后工作区成员只有我 1 人，**没有任何数字员工入区**。只有 5 个首批试点场景绑定了员工。❌
  - 附：立项后 `GET /api/scenarios` 列表不回写 workspace_id，场景与工作区只靠工作区侧 scenario_id 单向关联。
  - 好的方面：工作区讨论区自动写入 2 条系统消息（"场景已立项，项目工作区自动创建"+立项说明），上下文留痕完整。
- 独立完成：立项能；"确认数字员工已入区"无法完成——根本没入。

### ③ 进工作区 Agent 区 @数字员工派活 — ⚠️
- 预期：在 Agent 执行区 @ 对应数字员工派活。
- 实际：工作区#7 成员里无数字员工，但输入框提示"@ 可唤起数字员工候选"。我 @ 了董事办名下唯一的「财务经营数字员工」（DE-DSB-01）发日报需求，`POST /api/workspaces/7/messages` 返回 200，dispatched 生成任务 #8，秒级产出交付物，系统消息提示"待人工审核（审核人：师圆圆）"。链路通，但有两个问题：
  1. **@ 派发不校验工作区成员身份**——员工不在区里也能被@进来干活，那"立项拉员工入区"这一步的意义就被架空了；
  2. **我不知道该@谁**：场景没绑员工时，界面没有任何"该场景可用员工"提示，我是靠查 API 才知道董事办有个财务经营数字员工。普通 coach 会卡在这一步。
  3. 交付物是通用模板文案（"信息采集：完成，来源 3 处"），未体现真实经营数据——模拟环境可接受，但别当真实能力宣传。
- 独立完成：我能（懂技术、会试）；非技术同事大概率卡在"该@谁"。

### ④ 任务中心审核：驳回一次观察重做，再通过 — ⚠️
- 预期：驳回→数字员工按意见重做→再通过。
- 实际：
  - 任务中心是五列看板（待处理/进行中/待审核/已通过/已驳回），待审核 5 条，卡片有"点击审核 →"。证据：`shiyuanyuan_tasks.png`。
  - `POST /api/tasks/8/review {action:reject, comment:"内容太笼统…请补充后重做"}` → 200，工作区立刻出现系统消息"任务 #8 已按驳回意见重做，新交付物待人工审核"和一条新交付物。流程痕迹 ✅。
  - **但重做的交付物与第一版逐字相同**（同为"来源 3 处/记录 5 条/建议 2 条"模板），驳回意见根本没进重做上下文。❌ 形式上有重做，实质没有。
  - 再通过 {action:approve} → 200，状态"已通过"，done_at 落库，讨论区自动发通过通知。✅
  - 另：creator=reviewer=3，自己派活自己审，治理上缺"审核人不能是发起人"的约束（试点场景任务同理）。
- 独立完成：能，2 步搞定；但"重做"名不副实。

### ⑤ 治理中心申报火花奖 — ⚠️（UI 顺，API 不直觉）
- 预期：治理中心申报一个火花奖。
- 实际：治理中心"立项与激励"tab 右上「+申报激励」，列表已有金扳手/银齿轮/火花奖样板数据。我用 API 申报：首次按直觉传 `nominee_id` 被 400 拒绝（"nominee 必填"），改传 `nominee:"师圆圆"` 后 200，记录 #5 火花奖 800 元，状态"申报中"，列表核对在案。✅ 但字段命名不直觉；且激励记录无 title 字段，理由一段话了事。
- 独立完成：UI 走 1 步（点按钮填三项），能；API 第一次失败。

## 五、学习门槛评分与理由

**4 / 5**。左侧 10 个视图命名全是大白话（驾驶舱、场景库、任务中心……），场景库一键立项、工作区输入框自带 @ 提示、任务看板状态自明，文案无技术黑话，绝大多数操作 ≤3 步。扣 1 分：立项后没有"下一步去哪"的引导，动态流藏得太深，未绑员工的场景派活时会"不知道该@谁"。
**是否愿意每天用：愿意**。驾驶舱+任务中心正好覆盖我每天早上"看数、审活"两件事，心跳动态流可当晨会素材；前提是问题 1、2 修复，否则我每天面对的是断链和假重做。

## 六、总体判定

**修复后可落地**。「立项→派活→审核→激励」四段链路 API 与 UI 均走得通、留痕完整，框架是对的；但问题 1（立项不绑员工，阻塞扩围主流程）、问题 2（驳回重做空转）、问题 3（投入构成口径与方案不符）必须先修，否则平台只能演示、不能推广。
